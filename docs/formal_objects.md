# Formal Object Inventory for DeployStega

This document enumerates all formal objects used in the DeployStega system
and security definitions. Every object listed here corresponds to a
well-defined entity referenced in the paper and is required to have
unambiguous semantics in both the theoretical model and implementation.

For each object, we specify its type, role in the security experiment,
and whether it is chosen by the adversary, fixed by the model, or derived
algorithmically.

---

## Transcript (T)

**Type:** Abstract structured object

**Defined in paper as:**  
A covert transcript representing an intended pattern of covert communication,
consisting of a payload specification, a behavioral specification, and a
routing specification.

**Role in security experiment:**  
Represents adversarially specified intent for a hypothetical covert user. The
transcript is subjected to a feasibility check; if feasible, it is executed to
produce a synthetic interaction trace that replaces a benign user’s log in the
neighboring dataset.

**Chosen by:**  
Adversary

**Notes / Invariants:**  
- Contains no absolute timestamps  
- Contains no platform-specific identifiers  
- Does not directly reference logs, features, or observables  

---

## Payload Specification

**Type:** Component of Transcript

**Defined in paper as:**  
An abstract specification of the bitstring to be communicated covertly, defined
independently of any concrete token positions, timing realizations, or execution
outcomes.

**Role in security experiment:**  
Specifies the covert message to be conveyed. The message is fixed prior to
execution; the system determines when, where, and whether embedding occurs
subject to feasibility constraints.

**Chosen by:**  
Adversary

**Notes / Invariants:**  
- Does not reference tokens, probabilities, semantic features, or observables  
- Embedding may be skipped, delayed, or split if feasibility constraints apply  

---

## Behavioral Specification

**Type:** Component of Transcript

**Defined in paper as:**  
An abstract description of intended interaction structure and relative timing,
without absolute timestamps.

**Role in security experiment:**  
Defines adversarial intent over behavioral structure; concrete timings are
generated by the system to remain feasible.

**Chosen by:**  
Adversary

**Notes / Invariants:**  
- Specifies relative ordering and session structure only  
- Cannot reference logs, timing features, or observables  

---

## Routing Specification

**Type:** Component of Transcript

**Defined in paper as:**  
An abstract routing intent over artifact classes and relative access
relationships.

**Role in security experiment:**  
Defines adversarial intent over routing structure; concrete artifact identifiers
are resolved by the system.

**Chosen by:**  
Adversary

**Notes / Invariants:**  
- References artifact classes, not concrete identifiers  
- Cannot condition on routing features or access observables  

---

## Feasibility Region (R)

**Type:** Set of valid transcripts

**Defined in paper as:**  
The set of transcripts that satisfy empirically learned timing, message-format,
and routing constraints derived from benign platform traces.

**Role in security experiment:**  
Defines admissible adversarial behavior. Only transcripts in R may be executed;
transcripts outside R are rejected.

**Chosen by:**  
Model (learned from benign data)

**Notes / Invariants:**  
- Fixed prior to evaluation  
- Deterministic membership test  
- Independent of adversarial choices  

---

## Feasibility Check (Feasible(T))

**Type:** Boolean-valued function

**Defined in paper as:**  
A deterministic feasibility check returning true if T is in R and false
otherwise.

**Role in security experiment:**  
Determines whether an adversarially chosen transcript is admissible.

**Chosen by:**  
Model

---

## Execution Function (Execute(T))

**Type:** Deterministic mapping function

**Defined in paper as:**  
A function that maps a feasible transcript T to an executable interaction trace
consistent with benign platform behavior.

**Role in security experiment:**  
Produces the synthetic interaction log that replaces a benign user’s log in the
neighboring dataset D′.

**Chosen by:**  
Model

**Notes / Invariants:**  
- Only defined for feasible transcripts  
- Produces observables restricted to capability class F  

---

## Capability Class (F)

**Type:** Set of feature-extraction functions

**Defined in paper as:**  
A fixed set of deterministic functions, each of which maps raw platform
interaction logs to finite sets of semantic, behavioral, and routing feature
values.

**Role in security experiment:**  
Defines the adversary’s observational interface. All adversarial classifiers
operate exclusively on features produced by functions in F.

**Chosen by:**  
Model

**Notes / Invariants:**  
- Each feature extractor operates over a fixed scope (e.g., actions, sessions,
  users, or artifact-level histories)  
- F bounds the adversary’s information  

---

## Semantic Feature Set (F_semantic)

**Type:** Subset of Capability Class

**Defined in paper as:**  
The subset of feature extraction functions in F whose outputs are derived from
token-level linguistic analysis.

**Role in security experiment:**  
Captures detectability arising solely from linguistic steganography.

**Chosen by:**  
Model

**Notes / Invariants:**  
- Deterministic feature extractors  
- Outputs finite sets of linguistic feature values  
- No behavioral or routing information is used  

---

## Behavioral Feature Set (F_behavioral)

**Type:** Subset of Capability Class

**Defined in paper as:**  
The subset of feature extraction functions in F whose outputs capture
application-level behavioral features.

**Included features:**  
- Intra-user timing gaps  
- Action frequency statistics  
- Temporal correlation structure  
- Review latency  
- Retrieval latency  

**Role in security experiment:**  
Captures detectability arising from user behavior and interaction patterns.

**Chosen by:**  
Model

**Notes / Invariants:**  
- Deterministic feature extractors  
- Each extractor declares its aggregation unit (actions, sessions, users,
  or user pairs)  
- No semantic or routing information is used  

---

## Routing Feature Set (F_routing)

**Type:** Subset of Capability Class

**Defined in paper as:**  
The subset of feature extraction functions in F whose outputs capture artifact
access topology and routing structure.

**Included features:**  
- Access topology patterns  

**Role in security experiment:**  
Captures detectability arising from routing behavior and access-path structure.

**Chosen by:**  
Model

**Notes / Invariants:**  
- Operates solely on access metadata and interaction sequences  
- No payload or semantic information is used  

---

## Dataset (D)

**Type:** Finite collection of interaction traces

**Defined in paper as:**  
A dataset of benign user interaction logs drawn from a real LLM-mediated
application environment.

**Role in security experiment:**  
Serves as the baseline dataset for indistinguishability testing.

**Chosen by:**  
Model

---

## Neighboring Dataset (D′)

**Type:** Finite collection of interaction traces

**Defined in paper as:**  
A dataset identical to D except that one (or k) user’s interaction trace is
replaced with a synthetic trace generated by DeployStega.

**Role in security experiment:**  
Used to test detectability of covert communication.

**Chosen by:**  
Model (replacement construction), adversary indirectly via choice of T  

---

## Adversary (A)

**Type:** Adversarial classifier

**Defined in paper as:**  
An entity that observes features from capability class F and attempts to
distinguish between datasets D and D′.

**Role in security experiment:**  
Attempts to detect the presence of covert communication.

**Chosen by:**  
The adversary selects any classifier whose inputs are limited to features
extractable by F.

---

## Indistinguishability Parameter (ε)

**Type:** Scalar metric

**Defined in paper as:**  
An empirical measure of adversarial advantage computed as  
ε = ln(TPR / FPR).

**Role in security experiment:**  
Quantifies detectability under capability class F.

**Chosen by:**  
Computed by the model during evaluation

---

## Population Size Parameter (k)

**Type:** Integer

**Defined in paper as:**  
The number of users replaced with covert users in population-level experiments.

**Role in security experiment:**  
Controls covert-user density.

**Chosen by:**  
Model

---

## Dead-Drop Resolution Function (DeadDrop(t))

**Type:** Deterministic mapping

**Defined in paper as:**  
Maps a shared PRNG seed and epoch to a specific application artifact.

**Role in security experiment:**  
Implements application-layer routing indistinguishable from benign access.

**Chosen by:**  
Model

---

## PRNG Seed (s)

**Type:** Bitstring

**Defined in paper as:**  
A shared seed used to synchronize dead-drop resolution.

**Role in security experiment:**  
Enables deterministic artifact selection.

**Chosen by:**  
Out-of-band assumption

---

## Epoch Schedule (t)

**Type:** Discrete time index

**Defined in paper as:**  
A shared epoch counter used to index dead-drop resolution.

**Role in security experiment:**  
Coordinates routing decisions over time.

**Chosen by:**  
Out-of-band assumption

---

## Feature Extractors

**Type:** Deterministic functions

**Defined in paper as:**  
Functions in F that map interaction traces to finite sets of feature values.

**Role in security experiment:**  
Provide inputs to adversarial classifiers.

**Chosen by:**  
Model

---

## Adversarial Classifiers

**Type:** Binary classification algorithms

**Defined in paper as:**  
Classifiers selected and trained by the adversary to distinguish between sets of
features extracted from D and D′.

**Role in security experiment:**  
Operationalize detectability and define empirical ε.

**Chosen by:**  
Adversary (subject to capability class F)
