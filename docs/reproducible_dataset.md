# DeployStega Open Interaction Dataset

## Specification and Research Usage Guide

## 1. Purpose and Scope

The **DeployStega Open Interaction Dataset** is an interaction dataset intended to support empirical research on covert communication mediated by large language models (LLMs) under behavioral, routing, and semantic adversaries.

The dataset is motivated by the observation that, in realistic enterprise and platform deployments, adversaries primarily observe application and platform logs (e.g., audit logs, access telemetry, routing metadata). Accordingly, the dataset represents adversary-visible interaction traces, capturing when, how, and which artifacts users interact with, while supporting linkage to semantic content through stable references.

The dataset is designed to enable rigorous evaluation of:

- **Behavioral-only adversaries** operating exclusively on interaction logs,
- **Semantic-only adversaries** operating on content artifacts (when available), and
- **Cross-layer adversaries** combining behavioral and semantic signals.

Our dataset supports the fact that semantic-only steganalysis is supotimal in realistic logging regimes; behavioral filtering can substantially reduce the cost of semantic inspection by narrowing attention to a small subset of suspicious interactions.

---

## 2. Dataset Delivery Model

### 2.1 What external researchers receive

External researchers are provided with a complete, static dataset release. They are not required to run any code from the DeployStega repository.

The dataset release includes:

- One or more JSONL files containing interaction traces,
- Trace-level labels (`benign` or `covert`),
- Semantic artifact references, when semantic content is included.

The dataset is intended to be consumed in the same manner as other released log or audit datasets, and is usable with standard data-loading and analysis tools.

---

## 3. Formal Data Model

The dataset follows a strict three-level hierarchy:

```
Dataset
└── Trace (exactly one per user)
    └── Event (time-ordered)
```

### 3.1 Event (atomic unit)

An event corresponds to a single interaction, analogous to an application log entry. Each event includes:

- **timestamp** (required): UNIX timestamp in seconds,
- **action_type** (required): categorical action label (e.g., `route_access`),
- **artifact_ids** (required): identifiers of artifacts referenced by the action,
- **metadata** (required): contextual key–value information visible in logs.

A canonical event record has the form:

```json
{
  "timestamp": 1713456123.0,
  "action_type": "route_access",
  "artifact_ids": [42],
  "metadata": {
    "artifact_class": "issue",
    "role": "sender",
    "semantic_ref": "issue_42_body_v3"
  }
}
```

### 3.2 Trace (one user)

A trace is a temporally ordered sequence of events generated by exactly one user. Traces do not mix users and preserve event ordering.

Each trace is labeled at the user level (see Section 6).

### 3.3 Dataset (many users)

A dataset is a collection of traces, one per user. In the released format, each line of the JSONL file corresponds to one user trace.

## 5. Semantic Content: Representation and Rationale

### 5.1 Why semantic text is not embedded by default

In realistic deployments, application and audit logs typically do not contain full message bodies. Embedding semantic content directly into interaction logs would therefore grant adversaries unrealistic visibility and undermine the intended threat model.

Consequently, the interaction dataset captures behavioral and routing information only.

### 5.2 Semantic references (semantic_ref)

When semantic analysis is required, events may include a `semantic_ref` field in metadata. This field is an opaque identifier that links the event to a separately released semantic artifact dataset (e.g., clean and steganographic text snapshots).

This separation enables:

- **Behavioral-only evaluation** using interaction logs,
- **Semantic-only evaluation** using text artifacts,
- **Cross-layer evaluation** by joining on `semantic_ref`.

## 7. Timestamps and Temporal Fidelity

The released dataset uses real UNIX timestamps derived from logged interactions. This enables faithful evaluation of:

- Intra-user timing distributions,
- Session segmentation and session lengths,
- Temporal correlation and access patterns.

## 8. Prescribed Research Use (Concrete Guidance)

External researchers are expected to use the dataset as follows:

1. Load the JSONL dataset using standard tooling.
2. Treat each line as a single user trace with time-ordered events.
3. Extract behavioral, routing, and occassioanlly semantic features directly from traces.
5. Evaluate detection performance and cost reduction by:
   - Measuring how many traces or events are flagged by behavioral detectors,
   - Comparing semantic-only detection against behaviorally filtered detection.

This procedure directly supports the dataset's primary research contribution: demonstrating that behavioral filtering reduces the cost of semantic steganalysis without sacrificing detection capability.
