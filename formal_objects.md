**\# Formal Object Inventory for DeployStega**

This document enumerates all formal objects used in the DeployStega system  
and security definitions. Every object listed here corresponds to a  
well-defined entity referenced in the paper and is required to have  
unambiguous semantics in both the theoretical model and implementation.

For each object, we specify its type, role in the security experiment,  
and whether it is chosen by the adversary, fixed by the model, or derived  
algorithmically.

**Transcript (T)**

**Type:** Abstract structured object

**Defined in paper as:**  
A covert transcript representing an intended pattern of covert communication,  
consisting of a payload specification, a behavioral specification, and a  
routing specification.

**Role in security experiment:**   
Represents adversarially specified intent for a hypothetical covert user. The transcript is subjected to a feasibility  
check; if feasible, it is executed to produce a synthetic interaction trace  
that replaces a benign user’s log in the neighboring dataset.

**Chosen by:**   
Adversary

**Notes / Invariants:**  

- Contains no absolute timestamps    
- Contains no platform-specific identifiers    
- Does not directly reference logs, features, or observables  

## **Payload Specification**

**Type:** Component of Transcript

**Defined in paper as:**  
 An abstract specification of the bitstring to be communicated covertly, defined  
 independently of any concrete token positions, timing realizations, or execution  
 outcomes.

**Role in security experiment:**  
 Specifies the covert message to be conveyed. The message is fixed prior to  
 execution; the system determines when, where, and whether embedding occurs  
 subject to feasibility constraints.

**Chosen by:**  
 Adversary

**Notes / Invariants:**

- Does not reference tokens, probabilities, semantic features, or observables  
  Embedding may be skipped, delayed, or split if feasibility constraints apply

**Behavioral Specification:**

**Type**: Component of Transcript

**Defined in paper as:**  
An abstract description of intended interaction structure and relative timing, without absolute timestamps.

**Role in security experiment:**  
Defines adversarial intent over behavioral structure; concrete timings are generated by the system to remain feasible.

**Chosen by:**  
Adversary

**Notes / Invariants:**

- Specifies relative ordering and session structure only.  
- Cannot reference logs, timing features, or observables.

**Routing Specification**

**Type**: Component of Transcript

**Defined in paper as:**  
An abstract routing intent over artifact classes and relative access relationships.

**Role in security experiment:**  
Defines adversarial intent over routing structure; concrete artifact identifiers are resolved by the system.

**Chosen by:**  
Adversary 

**Notes / Invariants:**

- References artifact classes, not concrete identifiers.  
- Cannot condition on routing features or access observables.

**Feasibility Region (𝓡)**

**Type:** Set of valid transcripts

**Defined in paper as:**    
The set of transcripts that satisfy empirically learned timing, message-format,  
and routing constraints derived from benign platform traces.

**Role in security experiment:**    
Defines admissible adversarial behavior. Only transcripts T ∈ 𝓡 may be  
executed; transcripts outside 𝓡 are rejected.

**Chosen by:**    
Model (learned from benign data)

**Notes / Invariants:**  

- Fixed prior to evaluation    
- Deterministic membership test    
- Independent of adversarial choices  

**Feasibility Check (Feasible(T))**

**Type:** Boolean-valued function

**Defined in paper as:**    
A deterministic feasibility check returning true if T ∈ 𝓡 and false otherwise.

**Role in security experiment:**    
Determines whether an adversarially chosen transcript is admissible.

**Chosen by:**    
Model

**Execution Function (Execute(T))**

**Type**: Deterministic Mapping Function

**Defined in paper as:**   
A function that maps a feasible transcript T ∈ 𝓡 to an executable interaction  
trace consistent with benign platform behavior.

**Role in security experiment:**    
Produces the synthetic interaction log that replaces a benign user’s log in  
the neighboring dataset D′.

**Chosen by:**    
Model

**Notes / Invariants:**  

- Only defined for T ∈ 𝓡    
- Produces observables restricted to capability class F  

**Capability Class (F)**

**Type:** Set of feature-extraction functions

**Defined in paper as:**  
A fixed set of deterministic functions, each of which maps raw platform  
interaction logs to finite sets of semantic, behavioral, and routing feature  
values.

**Role in security experiment:**  
Defines the adversary’s observational interface. All adversarial classifiers  
operate exclusively on sets of features produced by functions in F; no  
access to raw logs or auxiliary information is permitted.

**Chosen by:**  
 Model

**Notes / Invariants:**

- Each feature extractor is defined over a fixed scope, such as individual actions, sessions, users, or artifact-level interaction histories, and produces a deterministic summary over that scope.  
- F bounds the adversary’s information; security guarantees hold only with   
  respect to this class

**Semantic Feature Set (F\_semantic):**

**Type:** Subset of Capability Class

**Defined in paper as:**  
 The subset of feature extraction functions in F whose outputs are derived from  
 token-level linguistic analysis, including extractors computing KL divergence,  
 perplexity, semantic coherence, and next-token distribution statistics.

**Role in security experiment:**  
 Captures adversarial detectability arising solely from linguistic  
 steganography, independent of behavioral or routing features.

**Chosen by:**  
 Model

**Notes / Invariants:**

- Each element of F\_semantic is a deterministic feature extractor  
- Outputs are finite sets of linguistic feature values  
- No behavioral or routing information is used

## **Behavioral Feature Set (F\_behavioral)**

**Type:** Subset of the capability class 

**Defined in paper as:**  
 The subset of feature extraction functions in F whose outputs capture 							     behavioral features of application-level interaction logs. 

 Fbehavioral={ft, ffreq, fcorr, freview-latency, fretrieval-latency}

**Role in security experiment:**  
 Captures adversarial detectability arising from user behavior and interaction  
 patterns, independent of semantic or routing features.

**Chosen by:**  
 Model 

**Notes / Invariants:**

- Each element of F\_behavioral​ is a deterministic feature extractor  
- Each extractor explicitly specifies the unit of aggregation over which it  
- operates (e.g., individual actions, sessions, artifacts, users, or pairs of  
- users  
- No semantic or routing information is used

## **Routing Feature Set (F\_routing)**

**Type:** Subset of the capability class

**Defined in paper as:**  
 The subset of feature extraction functions in F whose outputs are derived  
 from artifact access topology patterns 

Frouting={faccess-topology}

**Role in security experiment:**  
 Captures adversarial detectability arising from routing behavior and access-path  
 structure, independent of behavioral or semantic features.

**Chosen by:**  
 Model (as part of the threat model’s declared capability class)

**Notes / Invariants:**

- Extractors operate solely on access metadata and interaction sequences  
- No payload content or semantic features are used

**Dataset (D)**

**Type:** Finite collection of interaction traces

**Defined in paper as:**   
A dataset of benign user interaction logs drawn from a real LLM-mediated  
application environment.

**Role in security experiment:**    
Serves as the baseline dataset for indistinguishability testing.

**Chosen by:**    
Model (constructed from benign data sources)

**Neighboring Dataset (D′)**

**Type:** Finite collection of interaction traces

**Defined in paper as:**    
A dataset identical to D except that exactly one (or k) user’s interaction  
trace is replaced with a synthetic trace generated by DeployStega.

**Role in security experiment:**    
Used to test whether the presence of covert communication is detectable.

**Chosen by:**    
Model (replacement construction), adversary indirectly via choice of T

**Adversary (𝒜)**

**Type:** Adversarial Classifier

**Defined in paper as:**   
An entity that observes features from capability class F and attempts to  
distinguish between datasets D and D′.

**Role in security experiment:**    
Attempts to detect the presence of covert communication.

**Chosen by:**  

The adversary selects an arbitrary classifier, so long as  
its inputs are limited to features extractable by capability class F.

**Indistinguishability Parameter (ε):**

**Type:** Single, real-valued metric

**Defined in paper as:**    
An empirical measure of adversarial advantage computed as  
ε \= ln(TPR / FPR).

**Role in security experiment:**    
Quantifies the detectability of covert communication under capability class F.

**Chosen by:**    
Computed by the model during evaluation

**Population Size Parameter (k)**

**Type:** Integer

**Defined in paper as:**    
The number of users replaced with covert users in population-level  
indistinguishability experiments.

**Role in security experiment:**    
Controls the density of covert users in dataset D(k).

**Chosen by:**    
Model (experiment parameter)

**Dead-Drop Resolution Function (DeadDrop(t)):**

**Type:** Deterministic mapping

**Defined in paper as:**    
A function mapping a shared PRNG seed and epoch to a specific application  
artifact to route covert communication.

**Role in security experiment:**    
Implements application-layer routing indistinguishable from benign access.

**Chosen by:**    
Model (given shared seed)

**PRNG Seed (s):**

**Type:** Bitstring

**Defined in paper as:**   
A shared seed used by sender and receiver to synchronize dead-drop resolution.

**Role in security experiment:**    
Enables deterministic artifact selection without direct coordination.

**Chosen by:**    
Out-of-band assumption (fixed prior to experiment)

**Epoch Schedule (t)**

**Type:** Discrete time index

**Defined in paper as:**    
A shared epoch counter used to index dead-drop resolution.

**Role in security experiment:**    
Coordinates routing decisions over time.

**Chosen by:**    
Out-of-band assumption

**Feature Extractors**

**Type:** Deterministic functions

**Defined in paper as:**    
Functions in the capability class F that map interaction traces to finite sets  
 of extracted feature values.

**Role in security experiment:**    
Provide the inputs to adversarial classifiers.

**Chosen by:**   
Model

**Adversarial Classifiers:**

**Type:** Binary classification algorithms

**Defined in paper as:**   
Classifiers selected and trained by the adversary to distinguish between sets  
of feature values extracted from dataset D and neighboring dataset D′.

**Role in security experiment:**   
Operationalize detectability and define empirical ε.

**Chosen by:**    
The adversary may choose any classifier; its inputs are limited to sets of  
features extractable by the capability class F.
